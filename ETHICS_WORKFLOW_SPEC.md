# 伦理递交流程规格

> 来源：真实 CRA 工作流。本文用于指导后续伦理递交模块开发。

## 产品判断

当前系统已经能生成 `CRA to PI` 和 `PI to EC` 两类递交信，但真实工作并不是“生成一封信”就结束。

真实对象应该是：**伦理递交包**。

一个递交包包含：

- 项目组发布的新文件或修正案。
- 需要递交到某中心伦理委员会的一组递交资料。
- `CRA to PI` 递交信。
- `PI to EC` 递交信。
- 邮寄、PI 签署、伦理签收、审查方式、批件/意见函/回执、缴费等后续闭环。

因此后续开发应从“递交信记录”升级为“伦理递交包管理”。

同时，伦理递交包必须与两个模块打通：

- **项目文件管理**：递交文件清单应从项目文件库选择，并保存递交时的文件版本快照。
- **财务管理**：会审/快审产生的伦理审查费应进入财务伦理费台账。

详见 `CROSS_MODULE_INTEGRATION_SPEC.md`。

## 真实业务流程

1. 项目组不定时发布项目层面新文件。
   示例：药品管理手册、中心实验室手册、方案修正案、知情同意书修正案。
2. CRA 接到新文件后，需要在规定时间内递交给各中心伦理委员会。
3. CRA 创建递交包，确认项目基本信息、中心伦理信息、PI 姓名/职务、递交文件清单、版本号、版本日期完整无误。
4. CRA 生成两类递交信：
   - `CRA to PI`
   - `PI to EC`
5. CRA 打印并签署 `CRA to PI` 递交信。
6. CRA 将递交资料邮寄或递交至中心。
7. CRC 找 PI 签署完成递交信。
8. CRC/中心将资料递交至伦理委员会。
9. 伦理委员会签署接收 `PI to EC` 递交信，并确定审查方式：
   - 会审
   - 快审
   - 备案
10. 若为会审或快审：
    - 后续跟踪伦理批件或意见函。
    - 通常涉及伦理审查费缴费，需与财务管理模块联动。
11. 若为备案：
    - 收集伦理签收的递交信。
    - 部分中心还需收集备案受理回执。

## 核心实体

### 1. 递交包 ethics_submission_packages

用于管理一次递交闭环。递交包必须是**中心级**对象：同一项目文件可以在 A 中心已完成备案、B 中心等待 PI 签署、C 中心会审后等待批件。

建议字段：

- `id`
- `project_id`
- `center_id`
- `package_title`：递交包标题，如“方案 V2.0 修正案递交”
- `source_type`：来源类型，如项目组新文件、方案修正案、知情修正案、安全性文件、其他
- `received_from_project_date`：CRA 接到项目组文件日期
- `required_submit_by`：要求递交截止日期
- `status`：流程状态
- `review_method`：伦理审查方式，会审/快审/备案/待确认
- `cra_to_pi_letter_id`
- `pi_to_ec_letter_id`
- `courier_company`
- `tracking_number`
- `cra_signed_date`
- `sent_to_site_date`
- `pi_signed_date`
- `submitted_to_ec_date`
- `ec_received_date`
- `ec_receipt_collected`：是否已收集伦理签收递交信
- `filing_receipt_collected`：是否已收集备案受理回执
- `approval_or_opinion_status`：批件/意见函状态
- `approval_or_opinion_date`
- `fee_required`：是否需缴费
- `fee_status`：未申请/已申请/已打款/已完成/不适用
- `notes`
- `created_at`
- `updated_at`

### 2. 递交文件清单 ethics_submission_items

建议复用或扩展现有 `ethics_letter_items` 的字段，但从产品概念上应归属于递交包。

文件清单不应完全手工录入。后续应从项目文件库选择文件，并在递交包内保存当时的名称、版本号、版本日期快照，避免项目文件更新后影响历史递交记录。

建议字段：

- `id`
- `package_id`
- `project_document_id`
- `doc_type_snapshot`
- `doc_name_snapshot`
- `version_snapshot`
- `version_date_snapshot`
- `copies`
- `required_original`：是否原件
- `notes`
- `created_at`

### 3. 递交信 ethics_letters

现有表继续保留，作为递交包中的可生成文件。

后续可增加：

- `package_id`
- `letter_status`：草稿/已生成/已打印/已签署/已归档
- `signed_date`
- `archived_file_path`

## 状态设计

### 递交包状态

建议使用以下状态：

- `draft`：草稿，信息未补齐
- `ready_to_send`：资料已准备，待 CRA 打印签署/寄出
- `sent_to_site`：已寄往中心
- `pi_signed`：PI 已签署
- `submitted_to_ec`：已递交伦理
- `ec_accepted`：伦理已签收/受理
- `under_review`：会审/快审审查中
- `approved_or_opinion_received`：已收到批件或意见函
- `filed`：备案完成
- `completed`：递交闭环完成
- `needs_followup`：需补充/需跟进

### 审查方式

- `pending`：待确认
- `meeting_review`：会审
- `expedited_review`：快审
- `filing`：备案

### 完成条件

备案完成：

- `review_method = filing`
- 已收集伦理签收递交信
- 如中心要求，已收集备案受理回执

会审/快审完成：

- `review_method = meeting_review` 或 `expedited_review`
- 已收到伦理批件或意见函
- 如需缴费，费用状态已完成或明确无需本系统跟踪

## UI 设计建议

### 伦理递交首页

不再只展示“递交信记录”，而展示“递交包记录”。

列表关键列：

- 递交包标题
- 项目
- 中心
- 状态
- 审查方式
- 要求递交截止日期
- 伦理签收日期
- 批件/意见函状态
- 缴费状态
- 操作

筛选：

- 项目
- 中心
- 状态
- 审查方式
- 是否逾期
- 是否需跟进

### 新建递交包向导

建议 4 步：

1. 选择项目/中心/来源文件类型/时限。
2. 维护递交文件清单。
3. 生成或关联 `CRA to PI` 与 `PI to EC` 递交信。
4. 确认递交包并进入跟踪。

### 递交包详情页

建议结构：

- 顶部：状态条、审查方式、逾期/风险提示。
- 基本信息：项目、中心、PI、伦理委员会、来源类型。
- 文件清单：文件名称、版本号、版本日期、份数。
- 递交信：两封递交信的生成/打印/签署状态。
- 递交流程：寄出、PI 签署、伦理递交、伦理签收。
- 结果跟踪：备案回执、批件、意见函。
- 财务联动：是否需缴费、费用状态、关联发票。

## 开发拆分

### Phase 1：递交包骨架

目标：从“递交信列表”升级为“递交包列表”，先管理流程状态。

范围：

- 新增递交包表和递交文件表。
- 新增递交包列表。
- 新增递交包创建/编辑表单。
- 保留现有递交信生成逻辑。

不做：

- 暂不上传批件/回执附件。
- 暂不做财务自动联动。
- 暂不接 AI。

### Phase 2：双递交信绑定

目标：一个递交包下同时管理 `CRA to PI` 和 `PI to EC`。

范围：

- 递交包详情中一键生成两封递交信。
- 两封信共享同一份文件清单。
- 每封信有生成/打印/签署状态。

### Phase 3：伦理接收和审查结果

目标：跟踪伦理签收、审查方式、批件/意见函/备案回执。

范围：

- 添加伦理签收日期。
- 添加审查方式。
- 添加批件/意见函状态。
- 添加备案签收/备案受理回执字段。
- Dashboard 提醒待跟进递交包。

### Phase 4：财务联动

目标：会审/快审触发伦理审查费跟踪。

范围：

- 递交包可标记需缴费。
- 关联财务模块费用条目。
- Dashboard 提醒待缴费、待发票。

## 与项目文件管理打通

### 设计原则

- 项目文件库维护文件类型、名称、版本、版本日期。
- 递交包从项目文件库选择本次要递交的文件。
- 递交包保存文件快照，而不是只保存引用 ID。
- 递交信 Word 中的文件清单来自递交包快照。
- 项目文件是项目级主数据；递交状态、伦理签收、审查方式、批件/回执、费用状态均按中心分别维护。

### 产品价值

- 避免重复手输文件名和版本。
- 减少版本号错误。
- 以后可反查某个文件版本递交了哪些中心。
- 新文件发布后，可快速看到哪些中心还没递交。

## 与财务伦理费管理打通

### 设计原则

- 伦理模块负责判断“是否需要缴费”和关联递交包。
- 财务模块负责金额、打款、发票、到账状态。
- 递交包详情只展示费用摘要，不维护完整费用台账。

### 触发规则

- 会审：默认提示创建伦理审查费记录。
- 快审：默认提示创建伦理审查费记录。
- 备案：默认不创建费用记录，但允许手动标记需缴费。

### 产品价值

- 避免会审/快审后忘记伦理费。
- 财务可以按中心、项目、递交包追踪费用。
- Dashboard 可以提醒待缴费、待发票、金额异常。

## 首个可开发 Story

### Story E-0：项目文件库骨架

目标：先建立文件主数据，后续递交包从这里选择文件，避免重复录入。

验收标准：

- 可以按项目新增文件。
- 文件包含类型、名称、版本号、版本日期、接收日期。
- 可标记是否需要伦理递交。
- 项目详情页能看到项目文件列表。

### Story E-1：递交包状态追踪

目标：让 CRA 能记录一个递交包从草稿到伦理接收/完成的状态，不再只保存递交信。

验收标准：

- 可以新建递交包。
- 可以维护来源类型、接收日期、要求递交截止日期、审查方式。
- 可以从项目文件库选择递交文件，并保存文件快照。
- 可以设置流程状态。
- 递交包列表显示状态、审查方式、是否逾期。
- 不影响现有递交信 Word 生成功能。

建议先做这个 Story。
