# 人员培训管理规格

> 目标：当项目文件更新后，系统能追踪哪些授权研究人员需要培训、谁已完成、培训证据是否收集。

## 产品判断

项目文件发布后通常触发两类动作：

1. 伦理递交：文件需要递交中心伦理委员会。
2. 人员培训：授权研究人员需要接受新版文件培训。

因此，项目文件库不仅要服务伦理递交，也要服务培训管理。

关键原则：项目文件是项目级主数据，只说明某个文件版本是否需要培训；培训进度必须按中心、按授权研究人员分别追踪。同一份新版方案在 A 中心可能全员完成，在 B 中心可能只有 PI 完成，在 C 中心可能尚未安排。

## 真实业务场景

项目组发布新文件，例如：

- 新版方案
- 新版知情同意书
- 药品管理手册
- 中心实验室手册
- 重要安全性文件

CRA 需要判断：

- 是否需要递交伦理？
- 是否需要培训授权研究人员？
- 哪些中心需要培训？
- 哪些授权人员需要培训？
- 培训日期是什么？
- 是否需要生成培训记录/签到表 Word？
- 培训证据是否已收集？
- 培训未完成是否影响中心启动、入组或稽查风险？

## 核心实体

### 1. 项目文件 project_documents

项目文件库需要增加与培训相关的字段：

- `requires_training`：是否需要培训
- `training_due_days`：接收文件后多少天内完成培训
- `training_scope`：培训范围，如全体授权人员、PI/Sub-I、药品相关人员、实验室相关人员、自定义

### 2. 培训计划 training_plans

用于表示“某个文件版本需要对某些中心/人员进行培训”。

建议字段：

- `id`
- `project_id`
- `project_document_id`
- `title`
- `training_type`：方案培训、知情培训、药品手册培训、实验室手册培训、安全性培训、其他
- `scope`：全体授权人员、指定角色、指定人员
- `due_date`
- `status`：草稿、进行中、已完成、需跟进
- `notes`
- `created_at`
- `updated_at`

### 3. 培训对象 training_records

用于记录每个研究人员是否完成培训。

建议字段：

- `id`
- `training_plan_id`
- `project_id`
- `center_id`
- `staff_id`
- `staff_name_snapshot`
- `staff_role_snapshot`
- `required`：是否必须培训
- `training_date`
- `evidence_collected`：培训记录/签到表是否已收集
- `evidence_file_path`
- `status`：未开始、已完成、无需培训、需跟进
- `notes`
- `created_at`
- `updated_at`

### 4. 培训记录模板 training_templates

用于生成培训记录、培训签到表或培训确认表 Word。

建议字段：

- `id`
- `name`
- `template_type`：培训记录、签到表、培训确认表、其他
- `file_path`
- `is_default`
- `description`
- `created_at`

## 培训记录 Word 生成

### 目的

培训管理不能只停留在系统里勾选“已完成”。CRA 实际工作中需要生成可打印、可签署、可归档的培训记录或签到表。

### 生成对象

建议支持两种生成方式：

1. 按培训计划生成一份汇总培训记录 Word。
2. 按中心生成培训签到表 Word。

### Word 内容

培训记录模板至少应支持以下信息：

- 项目全称
- 项目简称
- 方案编号
- 中心名称
- PI 姓名
- 培训文件名称
- 文件版本号
- 文件版本日期
- 培训类型
- 培训日期
- 培训讲者/培训人
- 培训对象清单
- 研究人员姓名
- 研究人员角色/授权职责
- 签名栏
- 备注

### 模板占位符建议

- `{project_full_name}`
- `{project_name}`
- `{project_code}`
- `{center_name}`
- `{pi_name}`
- `{training_title}`
- `{training_type}`
- `{training_date}`
- `{trainer_name}`
- `{doc_name}`
- `{version}`
- `{version_date}`
- `{#trainees}{index}{/trainees}`
- `{#trainees}{staff_name}{/trainees}`
- `{#trainees}{staff_role}{/trainees}`
- `{#trainees}{training_date}{/trainees}`
- `{#trainees}{signature}{/trainees}`

### 证据闭环

培训 Word 生成后，流程应允许记录：

- 是否已打印
- 是否已签署
- 签署日期
- 是否已收集培训记录
- 归档位置或附件路径

早期版本可以只记录“证据是否收集”和“备注”，后续再做附件上传。

## 与研究人员管理打通

现有中心详情已有研究人员模块，培训管理应复用 `research_staff`。

培训计划创建时：

- 从中心研究人员列表中选择培训对象。
- 支持按角色筛选，例如 PI、Sub-I、药品管理员、CRC、实验室人员。
- 保存人员姓名和角色快照，避免后续人员信息变更影响历史培训记录。

## 与项目文件库打通

项目文件若标记 `requires_training = true`：

- Dashboard 应提示“新文件待创建培训计划”。
- 项目详情文件列表中显示培训状态。
- 文件详情中可查看关联培训计划。
- 文件详情中可从关联培训计划生成培训记录 Word。

## 与伦理递交打通

同一项目文件可能同时：

- 需要递交伦理。
- 需要研究人员培训。

页面上应能看到两个闭环：

- 伦理递交状态。
- 人员培训状态。

例如新版方案 V3.0：

- 伦理递交：A 中心已递交，B 中心待递交。
- 培训：A 中心 8/10 人已完成，B 中心 3/6 人已完成。

## Dashboard 提醒

Dashboard 后续应提醒：

- 新文件需要培训但尚未创建培训计划。
- 培训计划即将到期。
- 培训逾期未完成。
- 关键角色未完成培训，例如 PI、Sub-I。
- 培训证据未收集。
- 培训记录 Word 已生成但签署版未收集。

## 开发拆分

### Story T-1：项目文件增加培训标记

目标：登记项目文件时可标记是否需要培训。

验收标准：

- 项目文件可设置 `是否需要培训`。
- 可设置培训范围和建议完成天数。
- 项目文件列表显示培训标记。

### Story T-2：培训计划骨架

目标：基于某个项目文件创建培训计划。

验收标准：

- 可从项目文件创建培训计划。
- 可选择项目、文件、中心、培训类型、到期日期。
- 可从中心研究人员中选择培训对象。
- 可为每个对象生成培训记录。

### Story T-3：培训完成追踪

目标：追踪每位授权研究人员培训完成情况。

验收标准：

- 可标记单个人员已完成培训。
- 可记录培训日期。
- 可标记培训证据是否收集。
- 列表显示完成率和逾期人员。

### Story T-4：培训记录 Word 生成

目标：基于培训计划生成可打印签署的培训记录/签到表。

验收标准：

- 可上传或配置培训记录 Word 模板。
- 可从培训计划生成培训记录 Word。
- Word 包含项目、中心、文件版本、培训日期、培训对象和签名栏。
- 生成后可标记是否已打印、已签署、证据已收集。
- 生成逻辑复用递交信的 docxtemplater 能力。

### Story T-5：培训与 Dashboard 打通

目标：首页提醒培训风险。

验收标准：

- Dashboard 显示逾期培训。
- Dashboard 显示关键角色未培训。
- Dashboard 显示培训记录已生成但签署版未收集。
- Dashboard 可跳转到培训计划。
